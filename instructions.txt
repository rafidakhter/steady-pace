STEADY PACE V1 — CODEX BUILD INSTRUCTIONS
================================================

Purpose
-------
This document is the single source of truth for Codex to build the V1 React Native app.
Codex must follow the tasks in order, complete one micro task at a time, and keep the codebase scalable.

How Codex should work
---------------------
1. Read this entire document before writing code.
2. Follow phases in order.
3. Inside each phase, complete micro tasks sequentially.
4. Before any architecture or feature work, first create a basic Expo app that renders a simple "Hello World" screen and verify it runs successfully.
5. Do not skip architecture setup to rush UI after the Hello World verification step.
6. Keep business logic outside presentational components.
7. Keep the app data-driven so new sports, workouts, and plans can be added later.
8. Preserve clean structure and naming.
9. After each major phase, do a quick refactor pass before moving on.
10. If a task is unclear, infer the cleanest scalable solution rather than hardcoding.
11. Prefer reusable abstractions over one-off screen logic.

Project Summary
---------------
App name: Steady Pace

Platform:
- React Native
- Expo
- TypeScript
- iPhone + Android

V1 scope:
- Auth screens
- Challenge selection
- Home dashboard
- Weekly plan
- Workout detail
- Progress summary
- Local persistence

Pre-build verification requirement:
- Before building the real app structure, first make the Expo app run with a minimal Hello World screen
- Only proceed to the full architecture after confirming the base Expo app works

Design goals:
- Minimalist
- Editorial typography
- Mostly white and black
- Muted gray surfaces
- Green only for progress / active / success states
- Sharp corners
- Clean spacing
- Very little visual clutter

Scalability requirements:
- App must not be tied only to triathlon
- New sports must be easy to add later
- New workout types must be easy to add later
- New challenges and training plans must be easy to add later
- Training logic must be reusable
- Storage layer must be replaceable later

IMPORTANT IMPLEMENTATION RULES
------------------------------
1. Do not hardcode swim, bike, run directly into screen logic.
2. All sports must come from seed/config files.
3. All workout types must come from config/constants.
4. All plans must be data-driven.
5. UI components must not contain business rules.
6. Domain logic must live in services / utils / hooks / stores, not JSX.
7. Persistence must be abstracted so AsyncStorage can be replaced later.
8. Build a modular app that supports more phases and more sports later.
9. Use TypeScript everywhere.
10. Prefer composition over duplication.

SECTION P — ACCEPTANCE CRITERIA
================================================

V1 is complete when:

1. User can sign up locally.
2. User can sign in locally.
3. User can select a challenge.
4. Selected challenge persists after app restart.
5. Home screen shows today's workout from seeded plan logic.
6. Weekly plan shows current week's workouts.
7. Workout detail shows target values and lets user log actual values.
8. User can mark a workout complete.
9. Progress screen shows computed weekly totals and charts.
10. Workout logs persist after app restart.
11. Code is modular and scalable for future sports and plans.
12. The app runs on both iPhone and Android through Expo.
